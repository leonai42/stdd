## 各版本详细说明

### V1.0 — 初始版本（2026-05-03）

**背景**：STDD 方法论首次实现为可用的 CLI 工具和 Skill 系统。

**核心交付**：
- **6 阶段流程**：UNDERSTAND → SPEC → SLICE → BUILD → VERIFY → DELIVER
- **3 道强制确认门**：关键节点必须用户明确确认（Gate 1/2/3）
- **7 个 CLI 命令**：init、new、validate、status、archive、trace、install
- **6 个核心 Skill**：Markdown 指令，Claude Code 平台
- **模板系统**：proposal、design、spec、test-plan、tasks、slices、test-report
- **3 平台支持**：Claude Code、WorkBuddy、Trae
- **Python 开发规范** + 5 类初始失败模式检查

**里程碑意义**：STDD 从方法论概念变为可运行的工具。

---

### V1.1 — 长程开发（2026-05-06）

**核心交付**：
- **长程模式**：Phase 3-5 连续自动执行，一次性预授权
- **失败模式扩展**：从 5 类 → 9 类（新增 f/g/h/i）
- FPPT 项目实测中发现的 4 个 TDD 系统性盲区

**关键洞察**：用户确认是瓶颈——Gate 2 后的实现阶段可以自动化。

---

### V1.2 — 验证增强（2026-05-08）

**核心交付**：
- E2E 测试支持（可配置，Playwright 集成）
- 覆盖率诊断（不阻断，仅报告）
- 多 Python 版本兼容性测试
- 失败模式 9 类 → 11 类（新增 j/k：覆盖真空、契约断层）
- 基于 FPPT 项目验收测试回溯的 16 个实测问题

**关键洞察**：TDD 本身有盲区——静态结构正确 ≠ 运行行为正确。

---

### V2.0 — 架构升级（2026-05-13）

**核心交付**：
- **CLI 模块化**：687 行单文件 → `stdd/cli/` 包（8 模块 + 调度）
- **pytest 框架**：54 个测试用例，覆盖率 ≥ 80%
- **全局选项**：`--dry-run` 预览、`-v` 分级日志
- **新命令**：rollback、diff、abort（3 个）
- **Skill 系统标准化**：YAML frontmatter + 平台自动生成
- 新增 CHANGELOG.md、TROUBLESHOOTING.md、EXTENDING.md、examples/

**里程碑意义**：从"能跑的脚本"升级为"可测试、可维护的工程"。

---

### V2.3 — 基础配套完善（2026-05-18）

**核心交付**：
- **5 语言规范**：Python / Java / Go / Rust / TypeScript
- **6 平台支持**：+Cursor、Copilot、Aider（从 3→6）
- **配置模块化**：config.d/ 拆分（project/gates/long_range/quality）
- **Skill 标准化**：YAML frontmatter + _shared/ DRY 片段
- **开源准备**：README 中英分离、LICENSE、CONTRIBUTING.md

**里程碑意义**：STDD 从个人工具变为可开源项目。

---

### V2.4 — AI 辅助增强（2026-05-21）

**核心交付**：
- **自学习经验库**：`stdd experience` 5 子命令，11 类失败模式，自动加载/记录
- **Spec 自动补全**：`stdd extract-proposal` CLI + AI 生成 spec 草稿 + 置信度标签
- **智能切片推荐**：`stdd dependency-graph` + 五步分析（依赖→风险→工作量→分组→并行）
- **CI/CD 集成**：GitHub Actions + Pre-commit hook + PR comment 模板
- 新增 45 个测试（109 total）

**关键洞察**：项目级别的 AI 使用经验积累——这个能力是 STDD 区别于其他工具的核心。

---

### V2.5 — 经验生命周期闭环（2026-05-21）

**核心交付**：
- **经验 5 态 FSM**：discovered → verified → deposited → shared → merged/retired
- **社区经验共享池**：零后端设计（GitHub Releases + Gitee 镜像）
- **跨 Session 状态恢复**：resume_context 字段 + `stdd state`
- **Gate 文件确认**：token 文件 + CLI + 对话 三通道
- **非代码 Change 支持**：project_type 字段 + 替代检查清单
- **并行切片指南**：parallel_group 识别 + 子 Agent 派发
- 新增 46 个测试（155 total）

**里程碑意义**：经验从"项目私有"变为"社区共享"——网络效应开始。

---

### V2.7 — 工程化升级（2026-06-03）

**核心交付**：

**结构化基础**（吸收 V2.6）：
- Canonical YAML 格式（proposal/agent_spec/project-index）
- 双轨制文档基础（canonical/ 目录 + `stdd canon generate`）
- Spec 锚定法 L1-L4 方法论文档
- 7 个新 CLI 命令（proposal/canon/index/agent/hooks/structure/skill）

**锚定落地 + 双轨验证**：
- 锚定评估 Phase 2 集成 + Gate 2 检查
- 第 12 类失败模式 (l) 锚定缺失
- Agent 验证管线（CP 检查点 + 断言执行器）
- `stdd canon verify` 双轨一致性检查

**上下文工程**：
- phase-context.md 阶段交接摘要
- 上下文预算检查 + OpenCode 平台适配

**外部借鉴**：
- ECC 借鉴：Token 优化、Agent 细化、Hooks 增强、Skill 扩展
- CodeGraph 借鉴：代码结构摘要、经验 provenance、状态新鲜度、双语规则

**流程修复**（V2.7 复盘驱动）：
- 长程模式 7 条强制约束
- Step 1.4 每切片验证
- TC 实现覆盖检查 + 切片完成检查
- Gate 3 报告详报规范

**数据**：184 tests / 70% coverage / +29 tests from V2.5

**关键事件**：V2.7 开发过程中暴露了 STDD 自身的 25 个流程缺陷，驱动了 12 项流程修复。这是 STDD 方法论自我进化的重要里程碑。

---

### V2.8 — 质量度量升级（2026-06-03）

**核心交付**：

**质量度量**：
- pass@k 统计验证（k 次重复验证 + Spec 歧义自动检测）
- Plankton 多级自动修复（L1 静默/L2 建议/L3 报告）

**协作增强**：
- Two-Instance Kickoff（`stdd new --parallel`，双 Agent 并行启动）
- Rules 目录结构（`.stdd/rules/`，Phase 4 自动加载）

**遗留补齐**：
- 5 个 Skill 目录就位（python/fastapi/go/search-first/skill-create）
- 代码结构摘要完整实现（delta/merge/rebuild/show/graph）
- 覆盖测试补齐（10 模块覆盖达标或接近目标）

**流程改进**：
- build.md Step 1.5 并行切片合并验证
- design-adjustments 模板增加"影响 TC"字段
- build.md Step 1.6 phase-context 自动生成
- verify.md Step 2.1 非代码 Change 检测

**数据**：231 tests / 73% coverage / +47 tests from V2.5

## 每个版本的关键经验

| 版本 | 学到的关键教训 |
|------|---------------|
| V1.0 | 方法论需要工具支撑——纯文档无法执行 |
| V1.1 | 用户确认是瓶颈——长程模式是正确方向 |
| V1.2 | TDD 有盲区——静态正确 ≠ 动态正确 |
| V2.0 | 架构债必须尽早清理——模块化是测试的前提 |
| V2.3 | 开源需要多语言+多平台——单语言单平台无法推广 |
| V2.4 | AI 会反复犯同样的错——经验库是唯一解 |
| V2.5 | 经验网络效应——社区共享让所有人受益 |
| **V2.7** | **流程本身有缺陷——需要用 STDD 开发 STDD 来发现和修复** |
| **V2.8** | **乐观排期比流程欺骗更好——但要主动管理预期，不应闷头做到最后** |
