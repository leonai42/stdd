> 评审日期：2026-05-14

## 二、严重/高危问题

### [C1] 模板文件缺失 — 4/8 个模板不存在

**严重程度**：严重 **发现者**：文档审查

`STDD.md` 和 `README.md` 列出8个模板，但4个不存在：
- `tasks.md` — ❌ 不存在（slice.md Skill 尝试读取）
- `slices.md` — ❌ 不存在（slice.md Skill 尝试读取）
- `design-adjustments.md` — ❌ 不存在（verify.md Skill 尝试读取）
- `test-report.md` — ❌ 不存在（verify.md Skill 尝试读取）

**影响**：AI 执行 STDD 流程时，按 Skill 指令读取这些模板会失败，导致流程中断或生成格式错误的内容。

**修复建议**：
1. 创建缺失的4个模板文件
2. 更新 `STDD.md` / `README.md` 中的模板列表

---

### [H1] README.md 版本声明仍为 V1.4

**严重程度**：高 **发现者**：文档审查

`README.md` 第3行：`V1.4 · AI 辅助的...`，但当前版本为 V2.0。新命令（rollback、diff、abort）、config.d/ 架构、pytest 测试框架等 V2.0 特性均未在 README 中体现。

**修复建议**：更新 README.md 到 V2.0，补充新命令文档、config.d/ 架构说明、测试框架介绍。

---

### [H2] 平台 Skills 过期

**严重程度**：高 **发现者**：文档审查

`install.py` 重写后从核心 Skills（`.stdd/skills/`）生成平台 Skills。但 `.claude/skills/stdd-*/` 下的 Skills 仍是 V1.4 版本（5月3-9日），缺少 V2.0 的长程退出检测、CLI桥接检查、_shared/ 引用。

**影响**：用户通过 `/stdd-xxx` 使用的是旧版 Skill 指令，无法享受 V2.0 改进。

**修复建议**：重新执行 `stdd install claude-code` 更新已安装的 Skills。

---

### [H3] config.yaml 与 config.d/ 版本不一致

**严重程度**：高 **发现者**：配置审查

- `config.d/project.yaml`：`stdd_version: '2.0'`
- `config.yaml`：`stdd_version: "1.4.0"`

`read_config` 优先读取 config.d/，因此运行时版本为 2.0。但直接读取 config.yaml 的工具/脚本会看到 1.4.0。config.yaml 已成为死文件。

**修复建议**：删除旧的 `config.yaml` 或更新为与 config.d/ 一致的版本号。保留作为向后兼容备份时需明确标注。

## 四、低优先级问题

### 代码质量
- L1: `bin/stdd` 中 sys.path.insert(0, ...) 可能遮蔽 stdlib 模块
- L2: argv[0] 推断 STDD_SOURCE 的方式脆弱（依赖 bin/ 目录结构）
- L3: cmd_init 和 cmd_new 中模板缺失被无声跳过（应输出警告）
- L4: validate.py 中 AND 限制 (5) 硬编码为魔法数字
- L5: TC-ID 正则 `\d{3}` 严格限制3位数字，未来可能不够用
- L6: trace.py 和 diff.py 中 50行回看限制硬编码
- L7: diff.py 源目录硬编码为 `tests/` 和 `stdd/`，非标准目录被忽略
- L8: install.py 中安装后消息硬编码，无法为第三方平台定制
- L9: 跨命令的 logging 风格不一致（f-string vs %-formatting 混用）
- L10: `init.py` 使用 `getattr` 防御性访问但 `archive.py`/`abort.py` 直接访问属性
- L11: finder.py 依赖 mtime 排序，git 操作下不可靠

### 配置
- L12: read_config 使用浅合并，嵌套键冲突时后文件覆盖前文件
- L13: config.d/ 文件缺少原始 config.yaml 中的注释文档
- L14: 配置无 schema 验证，拼写错误无声忽略
- L15: read_config 无缓存，多次调用重复读取磁盘

### 文档
- L16: AGENTS.md 项目结构图过时（缺少 stdd/cli/、config.d/、_shared/）
- L17: AGENTS.md 包含绝对路径 `D:\mycode\FPPT\...` 不具可移植性
- L18: EXTENDING.md 中扩展点说明指向旧文件路径（`bin/stdd` vs `stdd/cli/commands/install.py`）
- L19: deliver.md Skill 描述手动归档流程，与 CLI `stdd archive` 命令重复

## 七、积极发现

在审查中发现的 V2.0 显著改进：

1. **模块化拆分成功**：单文件 687 行 → 15 个模块，每个 <150 行，可独立理解
2. **测试从零到 54 个**：首次引入自动化测试，81% 覆盖率
3. **API 设计一致**：所有命令遵循 `cmd_<name>(args: Namespace) -> None` 签名
4. **向后兼容良好**：`bin/stdd` 入口保留，所有旧命令签名不变
5. **配置拆分合理**：4 个职责单一的文件替代 103 行单文件
6. **Skill DRY 有效**：_shared/ 片段消除重复的确认门/模式选择/长程授权内容
7. **文档体系完善**：CHANGELOG、TROUBLESHOOTING、EXTENDING 首次引入
8. **示例项目完整**：hello-stdd 可独立初始化和运行，覆盖全部 6 阶段产出
9. **diff 命令创新**：spec↔test↔code 四列覆盖差异表，在其他工具中少见
10. **类型注解完整**：所有公开函数均有完整类型注解
