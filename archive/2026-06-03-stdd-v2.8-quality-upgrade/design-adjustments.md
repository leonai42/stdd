# V2.8 设计调整说明

> 记录 Phase 3-4 执行中对 Phase 2 设计的偏离和范围调整。
> **教训**：此文档应在 Phase 3 SLICE 阶段就创建，而非 Phase 5 事后补充。

## ADJ-1: 测试覆盖项（B3-B9）推迟到后续 patches

- **原设计**：B3-B9 共 7 个模块的覆盖率提升（index/experience/ci/proposal/agent/trace），在 V2.8 中全部完成
- **实际执行**：在 Phase 3 SLICE 时，Slice 2-3 的 workload 估算偏乐观（未区分新功能 vs 覆盖率补测试的工作量）。实际执行中优先完成了 A1-A6 新功能板块和 B1-B2 核心覆盖率提升，B3-B9 的覆盖测试未及实施
- **偏离类型**：Minor — 覆盖率提升是持续改进，不影响 V2.8 核心功能交付
- **影响**：index.py 50%、experience.py 72%、ci.py 73%、proposal.py 76%、agent.py 35%、trace.py 73% 的覆盖率暂未提升。这些模块的核心功能已有测试覆盖（Happy Path），缺失的主要是边界/错误路径
- **计划**：V2.8 后续 patches 或 V2.9 中补全

## ADJ-2: P2 流程改进（C1-C4）推迟

- **原设计**：C1-C4（并行切片合并验证 / 设计偏离 TC 追踪 / phase-context 自动生成 / 非代码 change 检测）在 V2.8 中实现
- **实际执行**：C1-C4 涉及 skill 指令和模板修改，每一项需要仔细设计措辞和交互逻辑，不适合快速实现。在有限时间内优先完成了高价值的新功能（A1-A6）
- **偏离类型**：Minor — P2 改进原本就是低优先级，推迟不影响 V2.8 核心价值
- **计划**：纳入 V2.9 或作为独立 change 推进

## ADJ-3: Phase 2 SPEC 设计深度不足（流程教训）

- **发现时间**：Phase 5 复盘（应为 Phase 2 即发现）
- **问题描述**：Phase 2 的 design.md 仅 1 页、specs 仅 3 个文件、test-plan 仅 15 个 TC。设计深度不足以支撑 19 项变更的完整实现。Phase 3 SLICE 未做 S/M/L 工作量标注，导致范围偏差在 Phase 4 才暴露
- **根因**：Phase 2 追求速度牺牲了完整性。如果能花 30 分钟（而非 10 分钟）做完整设计，就能在 Gate 2 时识别出"19 项中 A1-A6 是核心、B3-B9 是增量"的优先级差异，并主动提案调整
- **流程改进**：
  1. Phase 2 SPEC 应强制对每个 capability 做"可实现性评估"（不只是锚定评估）
  2. Phase 3 SLICE 的 S/M/L 工作量标注不应是可选的——每个切片必须有工作量估算
  3. **设计调整应在 Phase 3 首次创建**（而非 Phase 5）——一旦发现范围偏差，立即记录并提案
