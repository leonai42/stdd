## P0 — Bug 修复

### BUG-01：`stdd experience list` 新项目报错

- **现象**：`stdd experience list --language python --format json` → `FileNotFoundError: .experience-index.lock`
- **根因**：新项目 `.stdd/experiences/` 目录不存在，`_save_index` 尝试创建 lock 文件失败
- **修复**：`cmd_experience` list 操作时，experiences 目录不存在应优雅降级（返回空列表）
- **文件**：`stdd/cli/commands/experience.py`
- **验收**：新项目运行 `stdd experience list` 返回空列表而非报错

### BUG-02：`canon verify` DC-HASH 失败（source_hash 缺失）

- **现象**：`canon verify` 报告 DC-HASH 验证失败
- **根因**：Human View Markdown 先生成，此时 Canonical YAML 尚未生成，导致 source_hash 为空
- **修复**：调整生成顺序 — 先生成 Canonical YAML，再从 YAML 渲染 Human View MD，确保 source_hash 可回填
- **文件**：`stdd/cli/commands/canon.py`
- **验收**：`canon verify` 两项检查（DC-HASH + DC-FIELD）均通过

## P1 — 版本升级系统（Part A）

### UPGRADE-01：`stdd upgrade` CLI 命令

- **功能**：
  - `stdd upgrade` — 升级当前项目到最新版本
  - `stdd upgrade --check` — 仅检查版本差异
  - `stdd upgrade --all` — 升级所有注册项目
  - `stdd upgrade --all --check` — 查看所有项目版本矩阵
  - `stdd upgrade --lock` — 锁定当前项目版本
  - `stdd upgrade --unlock` — 解锁当前项目
  - `--yes/-y` — 跳过确认提示
  - `--dry-run` — 预览模式
- **升级流程**：读版本 → 比较 → 检查锁 → 备份 → 同步文件 → 重装平台 skills → 写 version.yaml → 更新注册表
- **文件**：`stdd/cli/commands/upgrade.py` (新)
- **依赖**：UPGRADE-02, UPGRADE-03

### UPGRADE-02：`.stdd/version.yaml` 项目版本文件

- **字段**：`stdd_version`, `locked`, `installed_at`, `upgraded_at`, `source_path`
- **逻辑**：首次 upgrade 时自动创建；旧项目回退读取 `project.yaml` 中的 `stdd_version`
- **文件**：`stdd/cli/utils.py`（`get_project_version()`、`_write_version_yaml()`）

### UPGRADE-03：`~/.stdd/projects.yaml` 全局注册表

- **字段**：`registry_version`, `projects[]` (path, version, locked, last_seen, name)
- **逻辑**：`stdd upgrade` 时自动注册；`--all` 遍历此表；stale 路径标记但不删除
- **文件**：`stdd/cli/commands/upgrade.py`（`_read_registry()`、`_write_registry()`、`_register_project()`）

### UPGRADE-04：启动版本检测

- **位置**：`stdd/cli/__init__.py` 的 `main()` 中，命令分发前
- **逻辑**：非 upgrade 命令 → 检查 `.stdd/` 是否存在 → 读取版本 → 与源版本比较 → 打印提示
- **条件**：锁定项目跳过；任何异常不阻塞
- **文案**：`[STDD] 有新版本可用: X.X.X (当前: Y.Y.Y)，运行 'stdd upgrade --check' 查看`

### UPGRADE-05：init.py 重构（提取共享常量）

- **内容**：将 `DIRS`, `FILES_TO_COPY`, `PLATFORMS` 提取为模块级常量，供 upgrade.py 复用
- **新增常量**：`CONFIG_MERGE_FILES`, `CONFIG_ALL_FILES`
- **文件**：`stdd/cli/commands/init.py`

## P3 — 轻量模式（Part B）

### LITE-01：复杂度评分模型

- **功能**：Phase 1 末尾，AI 基于 proposal 计算复杂度评分，建议模式
- **评分维度**：文件数(0-3) + 行数(0-3) + Capability数(0-2) + 风险等级(0-4) + 数据/API(0-2) + 安全(0-3)
- **阈值**：0-3→lightweight, 4-7→standard, 8+→thorough
- **文件**：`.stdd/skills/understand.md`

### LITE-02：模式标记 `.stdd.yaml`

- **字段**：`mode`, `task_type`, `complexity_score`, `score_confidence`, `mode_confirmed_by`, `mode_confirmed_at`
- **生命周期**：Gate 1 写入 → Phase 2 可上调 → Phase 2-6 按模式执行
- **文件**：`stdd/cli/commands/new.py`（扩展 `.stdd.yaml` 生成）

### LITE-03：Gate 1 模式选择

- **流程**：Phase 1 末尾 → AI 建议模式 → Gate 1 用户确认/调整 → 写入 `.stdd.yaml`
- **交互**：展示复杂度评分明细 + 推荐模式 + 各模式含义 → 用户选择
- **文件**：`.stdd/skills/understand.md`

### LITE-04：Phase 2-6 缩放表

- **内容**：每个 Phase 按 lightweight / standard / thorough 三档执行不同深度
- **设计原则**：
  - Phase 1 不缩放（统一流程，因为此时不知道模式）
  - Phase 2 SPEC：lightweight 跳过 design.md/test-plan/anchoring/canonical(仅proposal)
  - Phase 3 SLICE：lightweight 跳过切片分析
  - Phase 4 BUILD：lightweight RED 写 1-2 聚焦测试 + GREEN 最小实现 + 跳过 REFACTOR
  - Phase 5 VERIFY：lightweight 单 agent review + 核心 5 类失败模式 + 跳过 pass@k
  - Phase 6 DELIVER：lightweight 追加到批次而非独立归档
  - **TDD 底线保留**：轻量模式也必须 RED→GREEN
- **文件**：6 个 Skill 文件

### LITE-05：批次目录管理

- **策略**：monthly（默认）、weekly、count_based
- **目录命名**：`changes/_batch/{周期}-{创建日期}/`，如 `changes/_batch/2026-06-05/`
- **冲突处理**：同一天闭合再新建 → 追加小时后缀 `2026-06-05-14/`
- **配置**：`.stdd/config.d/lite.yaml`
- **文件**：新建 `stdd/cli/commands/batch.py`

### LITE-06：`stdd batch` CLI 命令

- `stdd batch list` — 查看当前批次状态（未闭合/已闭合）
- `stdd batch close` — 手动闭合当前批次
- `stdd batch status` — 查看当前批次详情（含 items 列表）
- **文件**：`stdd/cli/commands/batch.py` (新)

### LITE-07：非代码 task_type 支持

- **类型**：code, documentation, configuration, data-migration, dependency-upgrade
- **差异**：不同 task_type 对应不同验证策略和工具
- **共用**：共用轻量框架，仅验证步骤不同
- **文件**：`.stdd/config.d/lite.yaml` + `.stdd/skills/verify.md`

## P4 — CLI 改进

### CLI-01：`canon init` 默认目录改为 change 内部

- **现状**：默认在项目根目录创建 `canonical/`
- **目标**：默认在 `changes/<current-change>/canonical/` 下创建
- **选项**：`--per-change`（默认）/ `--project-level`（项目级持久规范）
- **文件**：`stdd/cli/commands/canon.py`
- **关联**：反馈问题 2

### CLI-02：`stdd structure delta/merge` 集成

- **内容**：Phase 4 BUILD 开始前自动执行 `stdd structure delta`；Phase 6 DELIVER 时 merge
- **文件**：`.stdd/skills/build.md` + `.stdd/skills/deliver.md`
- **关联**：反馈审查表 — 代码结构摘要未启用

### CLI-03：轻量模式自动检测

- **内容**：Phase 1 结束时，根据复杂度评分自动建议模式（无需用户手动判断）
- **依赖**：LITE-01, LITE-03
- **文件**：`.stdd/skills/understand.md`
