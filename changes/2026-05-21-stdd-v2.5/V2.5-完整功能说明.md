## 二、新增功能模块详解

### 模块 1：自学习经验库（V2.4 引入 + V2.5 闭环）

**V2.3 状态**：不存在经验库概念。

#### V2.4 — 基础建设

| 能力 | 说明 |
|------|------|
| `stdd experience list` | 按 category/language/lifecycle/severity 过滤列出经验 |
| `stdd experience add` | 添加经验条目（YAML frontmatter + Markdown body） |
| `stdd experience stats` | 经验库统计（按分类/语言/严重程度分布） |
| `stdd experience export` | 导出经验，自动脱敏（路径/IP/域名 → 占位符） |
| `stdd experience pull` | 占位符（V2.4 仅输出"V2.5 实现"） |
| 数据模型 | 经验文件 `.stdd/experiences/EXP-YYYY-NNNN.md`，11 类失败模式分类 |
| 自动索引 | `.experience-index.yaml`：按 category/language/lifecycle/severity 四维索引 |
| 技能集成 | `build.md` Step 0.5 加载经验；`verify.md` Step 3.5 自动记录经验 |

**11 类失败模式分类**：
```
(a) 幻觉行为    (b) 范围蔓延    (c) 级联错误    (d) 上下文丢失
(e) 工具误用    (f) 运行时偏差  (g) 管线断链    (h) 内容质量偏差
(i) 指令衰减    (j) 覆盖真空    (k) 契约断层
```

#### V2.5 — 生命周期闭环

**5 状态有限状态机**：

```
                    ┌──(export --publish)──→ shared ──(imported ≥3)──→ merged
                    │                              │                    │
discovered ──(verify)──→ verified ──(3+ occurrences)──→ deposited       │
    │                       │                              │            │
    └───────────────────────┴──(retire)────────────────────┴────────────┘
                                                              retired
```

| 新增子命令 | 说明 |
|-----------|------|
| `stdd experience verify <id>` | 手动确认经验有效性，discovered → verified |
| `stdd experience deposit <id>` | 沉淀经验，verified → deposited |
| `stdd experience retire <id> --reason "..."` | 废弃经验，任意状态 → retired |
| `stdd experience list --all` | 显示包括 retired 的全部经验（默认隐藏） |
| `stdd experience list --project-type python` | 按项目类型过滤 |

**自动提升逻辑**：
- `occurrences >= 2` 且 `confidence >= 0.7` → 自动提升为 `verified`
- `occurrences >= 3` 且 `confidence >= 0.8` → 自动提升为 `deposited`

**新增数据字段**（V2.5 扩展）：
```yaml
# 生命周期追踪
lifecycle_state: verified        # discovered|verified|deposited|shared|merged|retired
lifecycle_history:                # 状态变更时间线
  - state: discovered
    at: "2026-05-20"
  - state: verified
    at: "2026-05-21"
    by: "auto"                    # auto|manual

# 项目类型（防止跨类型经验污染）
project_type: python              # python|go|static_site|docs|config|null

# 社区投票（来自社区经验池）
community_votes_useful: 0
community_votes_unuseful: 0
adoption_count: 0
```

### 模块 2：Spec 自动补全（V2.4 引入）

**V2.3 状态**：用户手动编写 spec 全文，AI 不参与 spec 生成。

| 能力 | 说明 |
|------|------|
| `stdd extract-proposal` | 从 proposal.md 的 STDD-MARKER 注释中提取结构化数据，输出 JSON/YAML |
| `spec.md` 技能重写 | Step 2-3 替换为自动提取 → 自动生成 spec 草稿 → 置信度标签（✓高/⚠低） |
| 新增模板 `.stdd/templates/spec-draft.md` | AI 生成 spec 草稿的标准格式 |
| 用户角色转变 | 从"写 spec"变为"审 spec" |

**V2.5 扩展**：extract-proposal 提取字段从 4 个扩展到 8 个：

```json
// V2.4 提取（4 字段）
{
  "capabilities": [...],
  "dependencies": [...],
  "success_criteria": [...],
  "non_goals": [...]
}

// V2.5 提取（8 字段）
{
  "capabilities": [...],
  "dependencies": [...],
  "success_criteria": [...],
  "constraints": [...],       // 新增：技术/资源/时间约束
  "stakeholders": [...],       // 新增：干系人及其关注点
  "risk_areas": [              // 新增：结构化风险（含归因 capability）
    {"capability": "xxx", "risk": "xxx", "severity": "high"}
  ],
  "non_goals": [...],          // 新增：明确排除的范围
  "extracted_at": "2026-05-21T..."
}
```

### 模块 3：智能切片推荐（V2.4 引入）

**V2.3 状态**：用户手动拆分 capability 到切片，无自动化支持。

| 能力 | 说明 |
|------|------|
| `stdd dependency-graph` | 构建 spec 依赖图，输出 JSON/DOT/Text 三种格式 |
| 环检测 | 自动检测 A→B→C→A 循环依赖并报警 |
| 零依赖节点识别 | 标记可优先执行的独立 capability |
| 拓扑排序 | 按依赖关系排序执行顺序 |
| `slice.md` 技能增强 | 五步分析流程替代原 Step 2 |

**五步分析流程**：
```
Step 2.1: 依赖图分析 → dependency-graph 输出
Step 2.2: 风险评分   → 基于经验库匹配，对每个 capability 评分 1-5
Step 2.3: 工作量预估 → 按 complexity × dependency_depth 估算人天
Step 2.4: 智能分组   → 将相关 capability 分入同一 slice
Step 2.5: 并行化建议 → 标记 parallel_group，同组可并行执行
```

**slices.md 模板增强列**：

| risk | effort | parallel_group | rationale |
|------|--------|---------------|-----------|
| 3 | 1.5d | A | 无外部依赖，可优先 |

### 模块 4：CI/CD 集成（V2.4 引入 + V2.5 增强）

**V2.3 状态**：不存在 CI/CD 集成。

#### V2.4 — 基础 CI

| 命令 | 说明 |
|------|------|
| `stdd ci init` | 交互式生成所有 CI 配置文件 |
| `stdd ci generate workflow` | 生成 GitHub Actions workflow |
| `stdd ci generate pre-commit` | 生成 pre-commit hook 配置 |
| `stdd ci generate pr-template` | 生成 PR comment 模板 |
| `stdd ci check-failures` | 确定性失败模式检查（4 项，~60% 覆盖） |

**V2.4 的 4 项检查**：
```
(a) 文件存在性  → 检查 change 目录所有必需文件存在
(d) TC-ID 唯一性 → 检查 test-plan.md 中 TC-ID 无重复
(f) SHALL 关键字 → 检查所有 THEN 语句包含 SHALL
(g) AND 数量    → 检查 AND 数量 ≤ 5
```

#### V2.5 — 增强到 7 项检查（~80% 覆盖）

**架构重构**：引入检查注册表模式，从硬编码列表 → `@_register_check` 装饰器驱动的可扩展注册表。

```python
CHECKS = [
    check_files_exist,        # (a) 已有
    check_tcid_unique,        # (d) 已有
    check_shall_keyword,      # (f) 已有
    check_and_count,          # (g) 已有
    check_scope_creep,        # (b) 新增
    check_coverage_vacuum,    # (j) 新增
    check_contract_gap,       # (k) 新增
]
```

| 新增检查 | 失败模式 | 原理 |
|---------|---------|------|
| `ci check-scope` | (b) 范围蔓延 | `git diff --stat` vs proposal 声明的 capability 列表交叉验证 |
| `ci check-coverage` | (j) 覆盖真空 | 解析 `pytest --cov-report=json`，低于 quality.yaml 阈值（默认 80%）标记 |
| `ci check-contracts` | (k) 契约断层 | 跨 capability 依赖边字段一致性校验（基于 spec 的 GIVEN 模式匹配） |

**Graceful degradation**：无 `coverage.json` 时输出 `[SKIP]` 而非 `[FAIL]`，适用于非代码 change。

**3 个新子命令**：`ci check-scope`、`ci check-coverage`、`ci check-contracts` 可独立执行单项检查。

### 模块 5：社区经验共享池（V2.5 新增）

**V2.3/V2.4 状态**：经验仅在本地项目使用，无跨项目共享机制。

#### 架构设计：零后端

```
贡献侧                      维护者侧                      消费侧
───────                    ────────                     ────────

开发者 export --publish    维护者 curate pull/dedup     用户 pull python
       │                         │                            │
       │ PR 提交                  │ 全量拉取                    │ 主源: GitHub Release
       └──→ stdd-experiences ────┘→ 去重合并                    │ fallback: Gitee Release
           (GitHub 仓库)         逐条审核                       │ (5s 超时自动切换)
           (Gitee 镜像)          打包上传                       │
```

| 能力 | 说明 |
|------|------|
| `stdd experience pull <pack>` | 从社区池下载经验包，双源下载 + 5 秒超时自动 fallback |
| `stdd experience export --publish` | 脱敏 + 打包 tar.gz + lifecycle → shared，输出 PR 提交指引 |
| `stdd experience curate pull` | 维护者工具：全量下载经验提交到 inbox |
| `stdd experience curate deduplicate` | 维护者工具：Jaccard 相似度检测（>80% 自动合并，60-80% 标记） |
| `stdd experience curate review` | 维护者工具：交互式审核 [a]pprove/[e]dit/[m]erge/[r]eject/[s]kip |
| `stdd experience curate pack` | 维护者工具：构建官方 tar.gz 包 |

**配置段**（`.stdd/config.d/experience.yaml`）：
```yaml
community:
  registries:
    - name: github
      url: "https://github.com/leonai42/stdd-experiences/releases/latest/download"
      priority: 1
    - name: gitee
      url: "https://gitee.com/leonai42/stdd-experiences/releases/latest/download"
      priority: 2
  fallback_timeout: 5
  packs:
    - name: python
      version: "v1.0.0"
    - name: go
      version: "v1.0.0"
```

**投票机制**：GitHub Issues 作为投票 UI，经验 frontmatter 包含 `community_votes_useful/unuseful` + `adoption_count`。

### 模块 6：跨 Session 状态恢复（V2.5 新增）

**V2.3/V2.4 状态**：Agent 跨 session 后需翻找 change 目录才能恢复上下文。

| 能力 | 说明 |
|------|------|
| `stdd state` | 查看当前 resume context |
| `stdd state --resume` | 输出恢复提示（自然语言格式，可直接作为 AI prompt） |
| `stdd state --set active_slice 3` | 更新指定字段 |
| 自动写入 | 每个 phase 完成后自动更新 `last_action` / `last_modified` |

**数据模型**（`.stdd.yaml` 新增字段）：
```yaml
resume_context: "正在执行 STDD V2.5 的 BUILD 阶段，当前在第 3 个 slice（共 4 个）"
active_slice: 3
last_action: "完成了 ci-check-enhanced 的 TDD RED 步骤"
last_modified: "2026-05-21T15:30:00"
```

向后兼容：V2.4 的 `.stdd.yaml` 中这些新字段为 `null`，`read_resume_context()` 返回 `None`。

### 模块 7：Gate 文件确认通道（V2.5 新增）

**V2.3/V2.4 状态**：Gate 确认仅支持对话交互（AI 在对话中打印确认提示，用户回复"Yes"）。

V2.5 新增两种确认通道，实现去平台依赖：

| 通道 | 方式 | 说明 |
|------|------|------|
| `dialog` | 对话交互 | 原有方式，保留 |
| `file_token` | 创建 `GATE<N>_APPROVED` 空文件 | 新增，等效于确认 |
| `cli` | `stdd gate approve <change> --gate <N>` | 新增，等效于确认 |

三种通道底层统一操作：写入 `.stdd.yaml` 对应 phase 的 `confirmed_at` 时间戳。

`gates.yaml` 新增配置：
```yaml
confirmation:
  channels:
    - dialog
    - file_token
    - cli
  file_token_dir: "{change_dir}"
```

### 模块 8：并行切片执行指南（V2.5 新增）

**V2.3/V2.4 状态**：切片严格按顺序串行执行。

V2.5 在 `build.md` skill 中新增**并行执行策略**章节：

```
IF 存在 parallel_group 标记：
  1. 识别同 parallel_group 的切片
  2. 派发到多个子 agent 独立执行
  3. 主 agent 协调等待所有子 agent 完成
  4. 串行执行 REFACTOR 步骤（合并结果）
ELSE：
  按拓扑顺序串行执行（fallback）
```

纯 skill 指令增强，零 CLI 改动。要求 AI 平台支持 delegation（如 Claude Code 多 Agent 模式）。

### 模块 9：非代码 Change 支持（V2.5 新增）

**V2.3/V2.4 状态**：STDD 流程假设所有 change 都是代码类，verify 阶段的 11 类失败模式检查对纯文档/配置类 change 不适用。

#### 自动检测 project_type

经验创建时通过文件扩展名分布自动检测：
```
*.py       → python
*.go       → go
*.html/*.css/*.js → static_site
*.md       → docs
*.yaml/*.yml/*.toml → config
```

#### 经验加载过滤

`build.md` Step 0.5：只加载与当前 change 同 `project_type`（或 `project_type: null`）的经验。

#### 非代码替代检查清单

`verify.md` Step 3 后增加条件分支。当 change 目录不含 `*.py/*.go/*.java/*.rs/*.ts` 文件时，切换为 5 项检查：

| # | 检查项 | 说明 |
|---|--------|------|
| (a) | 链接有效性 | 所有外部链接可访问 |
| (b) | 文件范围一致性 | 变更文件与 proposal 声明一致 |
| (c) | 内部引用可达性 | 文档之间交叉引用路径正确 |
| (d) | 内容完整性 | 各章节覆盖所有声明的要点 |
| (e) | TC 目视验证覆盖 | 测试用例与 spec scenario 对应 |

## 九、关键架构决策

### 1. 经验状态机为何是 5 态而非 3 态

3 态（discovered → verified → retired）无法区分"本地验证通过"和"社区验证通过"。V2.5 的 5 态明确区分了：
- **verified**：单项目内验证通过
- **deposited**：同项目内多次复现，高置信度沉淀
- **shared**：已发布到社区（但未经验证被采纳）
- **merged**：被社区采纳 ≥3 次，成为官方经验

这样设计的原因是社区经验包的质量担保链路：只有 merged 状态的经验才能进入官方包，shared 状态需要经过 curate review。

### 2. CI 检查为何采用注册表模式

V2.4 的 `check-failures` 是硬编码 4 个函数调用，添加新检查需要修改函数体。V2.5 改为 `CHECKS` 列表驱动：

```python
# 添加新检查只需 3 步：
# 1. 写检查函数，用 @_register_check 装饰
# 2. check-failures 自动发现新检查
# 3. 添加独立子命令
```

好处：check-failures（聚合）和独立 check-* 子命令共享同一实现，不会出现"聚合报告和独立执行结果不同"的问题。

### 3. 社区池为何选择零后端

对比方案：
- **后端 API 服务**：需要服务器、数据库、运维，不符合 STDD 作为一个轻量 CLI 工具的定位
- **GitHub Releases + Issues**：零成本，利用 Git 生态已有的存储/CDN/投票机制，维护者操作也是普通 Git 操作

代价是实时性差（需手动 curate → pack → release），但对经验这种慢速变化的资产来说完全可接受。

### 4. project_type 为何不放在项目级 config

`project_type` 是经验条目级别的字段，而非项目级别。原因：
- 同一项目可能先后有不同类型的 change（基础设施用 Go，前端用 TypeScript）
- 经验加载时按"当前 change 的类型"过滤，而非"项目的固定类型"
