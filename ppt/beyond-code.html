05 / 12
V2.7 Technical Foundation

Agent 验证管线

agent_spec.yaml 格式

meta:
  task_id: deploy-staging
  system: production-server
steps:
  - id: CP-1
    description: 检查健康状态
    action: curl https://host/health
    assertions:
      - type: http_status
        expected: 200
      - type: stdout_contains
        expected: "ok"
rollback:
  steps:
    - docker compose down
    - docker compose up -d

🔧 支持断言类型

exit_code / stdout_contains / stderr_contains / http_status

🎯 设计哲学

机器可执行 + 人类可阅读 = 执行指令 & SOP 文档二合一

🔄 回滚支持

失败后不自动回滚，生成报告提示人工确认

Financial Risk Scenarios

为什么金融领域最适合 STDD？

传统痛点

风险1

策略上线靠人工 checklist，遗漏导致实盘损失

风险2

风控变更靠邮件审批，不可追溯不可审计

风险3

报表计算靠人工核对，疲劳导致遗漏

STDD 方案

方案1

agent_spec CP 自动验证 → 零遗漏

方案2

Gate 1/2/3 确认门 → 完整决策链

方案3

自动化验证 → 人只复核异常

金融是"零容错"领域 → STDD 的"定义预期→逐条验证"天然匹配

When to Use · When Not to Use

适用性边界

✅ 适用条件

可定义预期结果
可客观验证
有重复性（经验有价值）

⚠️ 需要适配

需扩展失败模式类别
需定制 agent_spec 模板
需建立领域术语表

❌ 不适用

纯创意工作
一次性任务
需实时直觉判断

核心判断标准：能否定义"做好是什么样"？
