## 二、V2.9 新增能力详解

### 2.1 三档执行模式（轻量 / 标准 / 全面）

**问题**：V2.8 的 6 Phase 流程对所有变更一视同仁——修一个 3 行的 bug 和架构迁移走同样的流程。用户对小变更倾向于跳过 STDD，导致代码失控。

**方案**：在 Phase 1 末尾根据 proposal 自动计算复杂度评分，建议执行模式。用户 Gate 1 确认后，Phase 2-6 按模式自动缩放。

**复杂度评分模型**：

```
评分 = 文件数(0-3) + 行数(0-3) + Capability数(0-2)
     + 风险等级(0-4) + 数据/API(0-2) + 安全(0-3)

0-3 → lightweight（微变更，1-3文件，<50行）
4-7 → standard（标准变更）
8+  → thorough（大型/关键变更）
```

**三档缩放对比**：

| Phase | lightweight | standard | thorough |
|-------|-------------|----------|----------|
| Phase 1 | 统一流程 | 统一流程 | 统一流程 |
| Phase 2 SPEC | 跳过 design/test-plan/anchoring | 标准 | 完整+置信度标签 |
| Phase 3 SLICE | 跳过（1隐式切片） | 智能切片 | 五步切片+并行化 |
| Phase 4 BUILD | RED(1-2聚焦测试)→GREEN，跳过REFACTOR | 标准TDD | 标准+pass@k |
| Phase 5 VERIFY | 单agent+核心5类失败模式 | 3agent+全12类 | 3agent+security+perf+pass@k=3 |
| Phase 6 DELIVER | 追加到批次 | 标准归档 | 完整+release notes |

**核心原则**：TDD 底线在任何模式下不妥协——轻量模式也必须写 RED→GREEN。差异在测试数量，而非有无。

**非代码任务支持**：通过 `task_type` 字段区分验证策略（documentation→markdownlint, configuration→yamllint, data-migration→dry-run, dependency-upgrade→pip-audit）。

### 2.2 版本升级系统

**问题**：STDD 源码升级到新版本后，各项目内的 skills/config 仍是旧版本。用户需手动逐个项目询问升级，不知道哪些项目是旧版本。

**方案**：完整的 CLI 版本管理系统。

**核心命令**：

```bash
stdd upgrade              升级当前项目
stdd upgrade --check      仅检查版本差异
stdd upgrade --all        批量升级所有已注册项目
stdd upgrade --all --check 查看全局版本矩阵
stdd upgrade --lock       锁定当前项目版本
stdd upgrade --unlock     解锁
stdd upgrade --dry-run    预览升级计划
```

**版本文件体系**：

| 文件 | 位置 | 用途 |
|------|------|------|
| `.stdd/version.yaml` | 项目级 | 版本号、锁定状态、安装/升级时间 |
| `~/.stdd/projects.yaml` | 用户全局 | 所有已知项目的版本注册表 |
| `.stdd/config.d/project.yaml` | 项目级 | 旧版兼容（无 version.yaml 时回退） |

**启动检测**：每次运行任意 `stdd` 命令时，自动检测项目版本与源版本差距。锁定项目静默跳过，有差距时打印一行提示。不阻塞任何操作。

**升级流程**：

1. 检测版本差距
2. 检查锁定状态
3. 备份当前文件到 `.stdd/backup/<old_version>-<timestamp>/`
4. 同步 skills / templates / standards / config 文件
5. `project.yaml` 使用合并策略（结构键覆盖 + 身份键保留）
6. 重新安装平台 skills
7. 写入 version.yaml + 更新全局注册表

### 2.3 批次目录管理

**问题**：轻量变更若每个创建独立 `changes/YYYY-MM-DD-xxx/` 目录，会导致目录爆炸（一个月60+个目录，每个只有2-3个小文件）。

**方案**：轻量变更共用批次目录，按时间窗口归档。

**目录结构**：

```
changes/_batch/
  2026-06-05/              ← 月度批次（6月5日首次创建）
    .stdd.yaml
    items/
      001-fix-typo/
      002-update-default/
    archive-summary.md
  2026-06-15/              ← 同上月度，但6月10日手动闭合后新建（不冲突）
```

**三种批次策略**：

| 策略 | 命名 | 适用场景 |
|------|------|---------|
| `monthly`（默认） | `YYYY-MM-DD` | 中小团队，月变更 <20 |
| `weekly` | `YYYY-Www-MMDD` | 中大团队，周变更频繁 |
| `count_based` | `batch-NNN` | 个人项目，变更不规律 |

**批次管理 CLI**：

```bash
stdd batch status   查看当前批次（策略、创建时间、item数）
stdd batch list     列出所有批次及状态
stdd batch close    手动闭合当前批次（生成 archive-summary.md）
```

**防冲突设计**：目录名用首次创建日期。同日闭合再新建 → 追加小时后缀（`2026-06-05-14/`）。天然防冲突，无需计数器。

## 八、V2.9 核心差异化

1. **三档执行模式**：从"一刀切"到"自适应"——修 bug 只需 3 步（意图→RED→GREEN），不浪费 token。TDD 底线不变。

2. **版本升级系统**：STDD 自身也纳入 STDD 管控。`stdd upgrade --all` 一键升级所有项目，锁定机制保护稳定期项目。

3. **批次目录管理**：轻量变更按月归档，避免 `changes/` 目录爆炸。三种策略适配不同团队规模。

4. **Skill 层彻底同步**：V2.8 的 CLI 能力（Canonical、Rules、Phase Context、Structure）终于被 Skill 指令引用，AI 不再"不知道有这些工具"。

5. **生产就绪**：TStrategy 项目试用反馈驱动的 2 个 bug 修复 + 12 项合规审查 → V2.9 是 V2.8 的"打磨版"。
