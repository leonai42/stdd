V2.3 完成了 STDD 的「基础设施」—— 5 语言规范 + 6 平台适配 + 配置模块化。V2.4 和 V2.5 在基础之上叠加了两层核心能力：**AI 辅助增强** 和 **多 Agent 适配**。

```

──────────────────── → ────────────────────── → ─────────────────────────
· 5语言开发规范          · 自学习经验库              · 经验生命周期闭环（5态FSM）
· 6大AI平台适配          · Spec自动补全              · 社区经验共享池（零后端）
· 模块化配置              · 智能切片推荐              · 跨Session状态恢复
· Skill标准化            · CI/CD集成                · Gate文件确认（去对话依赖）
· 11类失败模式检查        · 依赖图+环检测             · CI增强（scope/coverage/contracts）
                                                    · 多Agent并行切片派发
```

## 二、核心特性详解

### 特性 1：自学习经验库（V2.4 引入 + V2.5 闭环）

**解决什么问题**：AI 编程中最有价值的数据是「失败模式」—— 同一个坑，不同项目反复踩。V2.3 没有沉淀机制，每次犯错都是"第一次"。

**V2.4 基础建设**：经验数据模型（YAML frontmatter + Markdown body），11 类失败模式分类，自动索引（category/language/lifecycle/severity），`build.md` 在 Phase 4 开始前自动加载相关经验。

**V2.5 生命周期闭环**：

```
discovered ──(verify)──→ verified ──(3+ occurrences)──→ deposited
    │                          │
    │                          └──(export --publish)──→ shared ──(imported ≥3)──→ merged
    │                                                       │
    └───────────────────────────────────────────────────────┴──(retire)──→ retired
```

- **自动提升**：occurrences≥2 + confidence≥0.7 → verified；occurrences≥3 + confidence≥0.8 → deposited
- **新增命令**：`stdd experience verify/deposit/retire`
- **project_type 过滤**：经验按项目类型（python/go/static_site/docs/config）分类，防止跨类型经验污染

---

### 特性 2：跨 Session 状态恢复（V2.5）

**解决什么问题**：现代 Agent 平台（Hermes、Claude Code 多 Agent 模式）的核心特征之一是 **多 session、可中断、可恢复**。V2.3/V2.4 的 STDD 深度绑定单对话模式 —— Agent 重启 session 后需要翻找 change 目录下的 proposal/design/test-plan 等多个文件来重新理解上下文，通常需要 3-5 轮对话才能恢复进度。

**V2.5 方案**：`.stdd.yaml` 新增 4 个恢复字段，Agent 重启后读取即可在 **1 轮内**无歧义恢复：

```yaml
resume_context: "正在执行 STDD V2.5 的 BUILD 阶段，当前在第 3 个 slice（共 4 个）"
active_slice: 3
last_action: "完成了 ci-check-enhanced 的 TDD RED 步骤"
last_modified: "2026-05-21T15:30:00"
```

每个 phase 切换时自动更新这些字段。向后兼容：V2.4 的 `.stdd.yaml` 中这些新字段为 `null`。

**`stdd state` 命令**：
```bash
stdd state                          # 查看当前 resume context
stdd state --resume                 # 输出恢复提示（自然语言，可直接喂给 AI）
stdd state --set active_slice 3     # 更新指定字段
```

---

### 特性 3：Gate 文件确认 —— 去对话依赖（V2.5）

**解决什么问题**：V2.3/V2.4 的 Gate 确认唯一方式是"AI 在对话中打印提示 → 用户回复 Yes"。这在以下场景不适用：
- Hermes Agent 等非对话式执行平台（通过 API/CLI 调度，无交互对话窗口）
- CI/CD 流水线中自动确认（如低风险 change 自动通过 Gate 1）
- 批量操作（一次确认多个 change 的同一 Gate）

**V2.5 方案**：三种等效确认通道：

| 通道 | 方式 | 适用场景 |
|------|------|---------|
| `dialog` | 对话交互（原有） | 个人开发者日常使用 |
| `file_token` | 创建 `GATE<N>_APPROVED` 空文件 | Agent 平台通过脚本确认 |
| `cli` | `stdd gate approve <change> --gate <N>` | CI/CD 集成、批量操作 |

三种通道底层统一：均写入 `.stdd.yaml` 的 `confirmed_at` 时间戳，互相幂等。

```bash
# CLI 方式
stdd gate approve 2026-05-21-stdd-v2.5 --gate 1

# 文件方式（等价）
touch changes/2026-05-21-stdd-v2.5/GATE1_APPROVED
```

支持 Gate 顺序校验（Gate 2 必须在 Gate 1 之后确认）和重入幂等。

---

### 特性 4：多 Agent 并行切片（V2.5）

**解决什么问题**：V2.3/V2.4 的 Phase 4 BUILD 严格串行执行切片，即使多个切片之间零依赖。在支持 delegation 的 Agent 平台上（Hermes、Claude Code 多 Agent），这是对并行能力的浪费。

**V2.5 方案**：`build.md` skill 增加**并行执行策略**章节：

```
IF slices.md 中标记了 parallel_group：
  1. 识别同 parallel_group 的切片（零相互依赖）
  2. 派发每个切片到独立 subagent
  3. 主 agent 协调等待所有 subagent 完成
  4. 主 agent 统一执行 REFACTOR（提取共享逻辑、消除重复、全量测试）
ELSE：
  按拓扑顺序串行执行（自动降级，兼容无 delegation 能力的平台）
```

纯 skill 指令增强，零 CLI 改动。平台不支持 delegation 时自动 fallback 到串行。

---

### 特性 5：CI 检查增强 —— 60% → 80% 覆盖（V2.5）

**V2.4 基础**：`stdd ci check-failures` 覆盖 4 项结构检查（文件存在性、TC-ID 唯一性、SHALL 关键字、AND 计数）。

**V2.5 新增 3 项语义检查**：

| 检查 | 失败模式 | 原理 |
|------|---------|------|
| `ci check-scope` | (b) 范围蔓延 | `git diff --stat` vs proposal capability 声明的交叉验证 |
| `ci check-coverage` | (j) 覆盖真空 | 解析 pytest JSON 覆盖率报告，低于阈值（默认 80%）标记 |
| `ci check-contracts` | (k) 契约断层 | 跨 capability 依赖边的字段一致性校验 |

**架构升级**：从硬编码函数调用 → `CHECKS` 注册表模式（装饰器驱动），新增检查只需添加函数而不修改聚合逻辑。Graceful degradation：无数据时输出 `[SKIP]` 而非 `[FAIL]`。

---

### 特性 6：社区经验共享池（V2.5）

**零后端设计**：不引入任何服务器、数据库、API 服务。利用现有 Git 生态：

```
贡献者                     STDD 维护者                    使用者
───────                   ──────────                    ──────
export --publish          curate pull/dedup             pull python
      │                        │                            │
      └──→ stdd-experiences ──┘──→ 去重+审核+打包 ───→ GitHub Release（主源）
           (GitHub PR)                                      │
                                                    Gitee Release（镜像，5s超时自动切换）
```

- **双源下载 + 自动 fallback**：主源（GitHub Release）5 秒无响应 → 自动切 Gitee 镜像
- **投票机制**：GitHub Issues 作为投票 UI，经验条目包含 `community_votes_useful/unuseful` + `adoption_count`
- **维护者工具链**：`stdd experience curate pull/deduplicate/review/pack`

---

### 特性 7：非代码 Change 支持（V2.5）

V2.3/V2.4 的 verify 阶段默认所有 change 都是代码类，11 类失败模式检查对纯文档/配置类 change 不适用。

**自动检测**：根据文件扩展名分布判断 `project_type`（python/go/static_site/docs/config）。

**非代码替代检查清单**（5 项）：链接有效性 → 文件范围一致性 → 内部引用可达性 → 内容完整性 → TC 目视验证覆盖。

**经验隔离**：经验加载时只拉取与当前 change 同 `project_type` 的条目，防止跨类型污染。

## 三、Agent 框架适配详解

V2.5 的核心叙事之一是 **STDD 从「深度绑定单 AI 对话」升级为「跨 Agent 平台可移植」**。这一转变由 Hermes Agent 集成实战驱动。

### 3.1 问题诊断

V2.4 在 Hermes Agent 上运行时暴露出三个层面的适配缺口：

| 层面 | 问题 | 根因 |
|------|------|------|
| **交互** | Gate 确认依赖对话窗口，Agent 平台无此概念 | 确认通道单一（仅 dialog） |
| **状态** | Session 重启后 Agent 需要 3-5 轮重新理解进度 | 无结构化的状态恢复字段 |
| **并发** | 独立切片无法并行，多 Agent 能力闲置 | build.md 无并行指导（注：Hermes 本身支持 delegate_task batch 并行，但 V2.4 的 build.md 未指导如何利用） |

### 3.2 适配方案矩阵

Hermes Agent、Claude Code 多 Agent 和 OpenClaw 代表了三种典型的 Agent 平台范式，V2.5 同时适配：

| 适配能力 | Hermes（非对话式+delegation） | Claude Code 多Agent（对话式+delegation） | OpenClaw（多渠道+任务流） | 对应特性 |
|---------|---------------------------|--------------------------------------|--------------------------|---------|
| Gate 确认 | CLI 或文件 token（无需对话） | 对话 + CLI + 文件（三种均可） | CLI + 文件 token（渠道消息确认） | Gate 文件确认 |
| 状态恢复 | 读取 `.stdd.yaml` → 1 轮恢复 | 读取 `.stdd.yaml` → 1 轮恢复 | 读取 `.stdd.yaml` → 1 轮恢复 | 跨 Session 恢复 |
| 并行切片 | delegate_task batch（最多3并行） | 自动派发 subagent | Commander-Worker 派发 | 并行切片指南 |
| 批量操作 | `stdd gate approve` 脚本 | 对话中逐 change 操作 | Task Flow 调度 + CLI | Gate CLI + 文件 token |
| 经验加载 | 按 project_type 过滤 | 按 project_type 过滤 | 按 project_type 过滤 | 经验库 project_type |

### 3.3 平台兼容性总结

```
                     Hermes    Claude Code    OpenClaw    Cursor    Copilot    Aider     WorkBuddy    Trae
                     ──────    ───────────    ────────    ──────    ───────    ─────     ────────     ────
对话 Gate 确认         ✗              ✓            ✓          ✓         ✓         ✓          ✓          ✓
CLI Gate 确认          ✓              ✓            ✓          ✓         ✓         ✓          ✓          ✓
文件 Token Gate        ✓              ✓            ✓          ✓         ✓         ✓          ✓          ✓
跨 Session 恢复        ✓              ✓            ✓          ✓         ✓         ✓          ✓          ✓
并行切片派发           ✓              ✓            ✓          ✓         ✓         ✗          ✓          ✓
经验库加载             ✓              ✓            ✓          ✓         ✓         ✓          ✓          ✓
```

> 并行切片派发需要平台原生支持 subagent delegation。截至 2026 年 5 月，除 Aider（仅 Architect 串行模式）外的主流 AI 编程平台均已支持多 Agent 并行。各平台实现机制不同：Hermes delegate_task batch（最多 3 并行）、Claude Code 多 Agent 模式、OpenClaw Commander-Worker、Cursor /multitask（最多 4 并行）、Copilot Atlas 编排器、WorkBuddy Agent Teams、Trae 多 Agent 协同。不支持 delegation 的平台自动降级为串行执行，不影响功能完整性。

## 四、其他特性

### Spec 自动补全（V2.4）

- `stdd extract-proposal`：从 proposal.md 提取结构化数据（JSON/YAML）
- `spec.md` 技能重写：自动生成 spec 草稿 + 置信度标签（✓高/⚠低）
- V2.5 扩展：提取字段从 4 → 8（新增 Constraints / Stakeholders / RiskAreas / NonGoals）

### 智能切片推荐（V2.4）

- `stdd dependency-graph`：构建 spec 依赖图（JSON/DOT/Text），含环检测 + 拓扑排序
- 五步分析流程：依赖图 → 风险评分 → 工作量预估 → 智能分组 → 并行化建议

### CI/CD 集成（V2.4）

- `stdd ci init`：交互式生成 GitHub Actions workflow + pre-commit hook + PR template
- `stdd ci generate workflow/pre-commit/pr-template`

## 七、关键设计决策

### 经验状态机为何是 5 态而非 3 态

3 态（discovered → verified → retired）无法区分「单项目内验证」和「社区验证采纳」。5 态的设计用意：
- **verified**：单项目内确认有效
- **deposited**：同项目多次复现，高置信度
- **shared**：已发布到社区但未被采纳
- **merged**：被社区采纳 ≥3 次 → 可进入官方经验包

只有 merged 状态的经验才能进入官方包，shared 状态需要经过 `curate review`。

### 社区池为何选择零后端

对比后端 API 方案（服务器+数据库+运维），零后端方案利用 GitHub Releases（CDN）+ Gitee 镜像（国内可达）+ GitHub Issues（投票 UI），零成本且符合 STDD 轻量 CLI 工具定位。

### 并行切片为何不做 CLI

并行切片执行是 skill 指令层面的增强而非 CLI 命令。原因是：并行派发由 AI Agent 的 delegation 能力驱动，CLI 无法感知平台的 delegation 机制。纯 skill 指令也更容易跨平台兼容。
