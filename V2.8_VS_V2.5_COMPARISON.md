## 二、核心能力提升详解

### 2.1 双轨制文档体系（V2.7 新增）

**问题**：V2.5 的所有产出物是纯 Markdown——人类可读但机器不可解析。AI 消费 proposal/spec 时只能"阅读理解"，容易遗漏和误读。

**方案**：引入 Canonical YAML + Human View MD 双轨制。

| 能力 | V2.5 | V2.8 |
|------|------|------|
| proposal 格式 | 纯 Markdown | proposal.md (Human View) + proposal.yaml (Canonical) |
| spec 格式 | specs/*.md | specs/code/*.yaml (代码) + specs/agent/*.yaml (Agent) |
| 一致性保障 | 无 | `stdd canon verify` (源哈希 + 字段校验) |
| 文档生成 | 人工/AI 手动 | `stdd canon generate` (Canonical→Human View 单向生成) |
| 向后兼容 | — | canonical/ 目录可选，不存在时行为与 V2.5 完全一致 |

**新增 CLI**：`stdd proposal init/validate/show`、`stdd canon init/generate/verify`、`stdd index update/show/trace`

**价值**：为 V3.0 全自动生成引擎奠定基础。AI 消费结构化数据，人类阅读自然语言——各取所需。

---

### 2.2 Spec 锚定法（V2.7 新增）

**问题**：V2.5 的 spec 给了 AI 太多自由发挥空间。相同 spec 在不同 session 中可能产生不同实现——不可重复。

**方案**：四级锚定体系，在 Phase 2 SPEC 中评估 spec 自由度，确保关键 Change 的 spec 精确到"AI 没有发挥空间"。

| 锚定等级 | 方法 | 成本 | 适用场景 |
|---------|------|------|---------|
| L1 · 行为锚定 | THEN 用 SHALL 写死强制行为 | 无 | 所有 Change 默认 |
| L2 · 接口锚定 | 附加精确的函数签名/API 契约 | 低 | 多模块协作 |
| L3 · 模式锚定 | 引用参考实现模式 | 低 | 有成熟模式的重复工作 |
| L4 · 基准锚定 | 附参考实现代码 | 中 | 关键 Change、安全/金融 |

**新增失败模式**：(l) 锚定缺失 — critical Change 锚定等级不足时 Gate 2 阻断。

**价值**：从"事后检查 AI 有没有跑偏"升级为"事前不让 AI 有跑偏的空间"。

---

### 2.3 上下文工程（V2.7 新增）

**问题**：V2.5 的跨 session 恢复只有一句话指针（resume_context）。Agent 在新 session 中需要翻读 3-5 个文件才能重建上下文。

**方案**：

| 能力 | V2.5 | V2.8 |
|------|------|------|
| 阶段摘要 | 无 | phase-context.md（累积文件，每 phase AI 自动撰写） |
| 状态恢复 | 一句话 resume_context | 指针 + phase_context 引用 + 新鲜度校验 |
| 上下文管理 | 无指导 | build.md/verify.md Step -1 预算检查 |
| 平台覆盖 | 6 平台 | 7 平台（+ OpenCode，`stdd install opencode`） |

**价值**：新 session Agent 可在 1 轮 Read 内恢复完整上下文（vs V2.5 的 3-5 轮）。

---

### 2.4 经验库进化（V2.7 新增）

**问题**：V2.5 的经验条目只有单一置信度，不知道经验来自哪里。社区经验池只有架构设计，没有实际运营。

**方案**：

| 能力 | V2.5 | V2.8 |
|------|------|------|
| 经验溯源 | 无 | provenance 字段 (ci-detected/ai-inferred/human-reported/community-imported) |
| 权重体系 | 单一 confidence | provenance_weight × confidence 双重评估 |
| 自动升级 | occurrences 阈值 | provenance 升级 + occurrences 阈值 |
| 经验类别 | 11 类 | 14 类 (+agent_cp_failure, +spec_ambiguity, +anchor_missing) |
| 社区经验池 | 设计完成 | GitHub + Gitee 双仓库就绪，4 条种子经验 |
| 下载 fallback | — | 5s 超时 GitHub→Gitee 自动切换 |

**价值**：经验库从"静态分类"升级为"动态溯源"——知道经验从哪来，和知道经验是什么同等重要。

---

### 2.5 Agent 体系增强（V2.7+V2.8 新增）

**问题**：V2.5 的 review 只有 3 类 agent（code/test_config/docs_skills），缺少安全审计、性能分析等专项能力。Phase 2 没有独立 planner。

**方案**：

| 能力 | V2.5 | V2.8 |
|------|------|------|
| Subagent 数量 | 0（隐式） | 4 个（security/perf/compat/planner） |
| Review 维度 | 3 类 | 7 类（+security/performance/compatibility/architecture） |
| Agent 验证 | 无 | `stdd agent verify`（CP 检查点 + 断言执行器） |
| Hooks 自动化 | 无 | 3 个生命周期 hooks（session-start/pre-compact/stop）+ CLI |
| 双 Agent 启动 | 无 | `stdd new --parallel`（Two-Instance Kickoff） |

**价值**：从"通用 Agent 做所有事"升级为"专项 Agent 各司其职"。

---

### 2.6 质量度量升级（V2.8 新增）

**问题**：V2.5 的质量验证是"通过/失败"二元判断——缺少统计维度和自动修复能力。

**方案**：

| 能力 | V2.5 | V2.8 |
|------|------|------|
| 测试验证 | 单次运行 | pass@k（k 次重复验证，可配置 k=1/3/5） |
| Spec 歧义检测 | 无 | pass@1 低但 pass@3 高 → 自动标记 |
| 代码修复 | 无 | Plankton 3 级修复（L1 静默/L2 建议/L3 报告） |
| 修复 CLI | — | `stdd fix --level 1|2|3` |

**价值**：从"测一次"到"测 k 次"——统计上更可靠。从"只发现问题"到"自动修复简单问题"——效率提升。

---

### 2.7 工程效能优化（V2.7 新增）

**问题**：V2.5 没有 token 预算意识，没有文件大小约束，阶段间没有上下文重置指导。

**方案**：

| 能力 | V2.5 | V2.8 |
|------|------|------|
| 模型选择 | 无指导 | 6 阶段模型分层建议（Haiku→Sonnet→Opus） |
| 上下文重置 | 无 | 阶段间 `/clear` 提示 + 预算检查 |
| 文件大小 | 无约束 | 各语言规范 ≤500 行建议 |
| Skill 生态 | 6 个核心 Skill | +5 个专项 Skill（python/fastapi/go/search-first/skill-create） |
| Rules 体系 | 仅 standards/ | +`.stdd/rules/`（common/+5 语言，Phase 4 自动加载） |
| 代码结构知识 | 每 change 从零扫描 | 代码结构摘要系统（delta→index 自积累） |

**价值**：token 成本感知 + 项目越久越聪明（代码结构摘要累积效应）。

---

### 2.8 流程健壮性（V2.7+V2.8 修复）

**问题**：V2.7 开发过程中暴露了 STDD 流程自身的多个缺陷（V2.7 复盘识别 25 个缺陷点）。

**方案**：

| 修复项 | 描述 |
|--------|------|
| 长程模式 7 条强制约束 | 明确"跳过授权≠跳过流程"，每 Step 必须执行 |
| Step 1.4 每切片验证 | Phase 4 BUILD 每个切片完成后强制 TC 覆盖 + 产出物核对 + 测试运行 |
| 降级条件扩展 | 新增"切片 TC 覆盖率为 0"、"连续 3 切片无新增测试"、"产出物为 [TODO]" |
| Gate 3 报告制 | 未完成项必须逐项列出名称/原因/影响/补完计划，不得仅给计数 |
| TC 实现覆盖检查 | ci.py 新增 check_tc_implementation_coverage（test-plan vs 实际测试对账） |
| 切片完成检查 | ci.py 新增 check_slice_completion（done 标记必须有验证证据） |
| 设计调整规范 | Phase 3 就应创建设计调整文档（不应拖到 Phase 5） |
| 双语规则 | 10 条强制性约束中英对照，确保不同语言模型都能正确理解 |

**价值**：V2.8 的流程健壮性比 V2.5 显著提升——长程模式不再是"盲飞"模式，每切片有验证，Gate 3 报告信息完整，测试覆盖可追溯。

## 五、特色与优势总结

### V2.8 的核心差异化

1. **双轨制文档体系**：AI 消费 YAML，人类阅读 Markdown——各取所需，互不干扰。这是市面上其他 AI 编程工具没有的能力。

2. **Spec 锚定法**：从事后检查 AI 有没有跑偏，升级为事前不让 AI 有跑偏的空间。四级锚定适用于从简单修复到金融级安全的不同场景。

3. **自学习经验库进化**：从 V2.5 的"静态分类"升级为 V2.8 的"动态溯源"——provenance 追踪让经验可信度透明化。社区经验池（GitHub+Gitee 双仓库）让经验跨项目复用。

4. **质量度量双保险**：pass@k 统计验证 + Plankton 多级自动修复——既能量化质量，也能自动修复简单问题。

5. **流程健壮性**：V2.7 复盘驱动的 12 项流程修复，让长程模式从"盲飞"变为"可控自动驾驶"——每切片验证、Gate 3 详报、TC 覆盖可追溯。

6. **Agent 专业化**：从"一个通用 Agent 做所有事"到"4 个专项 Agent 各司其职"——安全审计、性能分析、兼容性检查、架构规划。

7. **项目越久越聪明**：代码结构摘要自积累 + 经验库自学习 + Rules 自动加载——三个机制让 STDD 随着项目成长变得越来越了解你的代码。
