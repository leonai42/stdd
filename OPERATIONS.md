## 一、核心定位与信息屋

### 一句话定位

> **STDD — 让 AI 写出正确代码的研发流程系统。不靠运气，靠系统。**

### 信息屋（Message House）

```
┌─────────────────────────────────────────────────┐
│  核心主张：AI 编程的质量保障体系                    │
│  Core Message: Quality Assurance for AI Coding   │
├─────────────────────────────────────────────────┤
│                                                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────────┐      │
│  │ 信任理由1 │  │ 信任理由2 │  │ 信任理由3    │      │
│  │ Spec先行  │  │ TDD兜底  │  │ 11类失败检查 │      │
│  │ 消灭歧义  │  │ 测试通过  │  │ 堵住AI常见错 │      │
│  └─────────┘  └─────────┘  └─────────────┘      │
│                                                  │
│  ┌─────────────────────────────────────────┐    │
│  │  实证：FPPT 5天从0到发布 · 27,826行 · 100%通过率 │    │
│  │        TStrategy V4.2 · 19,500+测试行         │    │
│  └─────────────────────────────────────────┘    │
│                                                  │
│  ┌─────────────────────────────────────────┐    │
│  │  兼容：6平台 · 5语言 · MIT开源 · 零外部依赖  │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

### 目标用户画像

| 用户类型 | 痛点 | STDD 卖点 |
|---------|------|----------|
| **AI 编程实践者**（开发者） | AI 写的代码有 bug，跑偏，不敢信任 | 11 类故障检查 + 测试兜底，AI 翻车率大幅降低 |
| **技术 Leader / 架构师** | 想推 AI 辅助开发但担心质量失控 | 3 道确认门 + 双向追溯，管理可控 |
| **开源维护者** | AI 提的 PR 不可追溯、质量参差 | Spec 先行 + TDD，贡献质量有保障 |
| **金融/医疗 IT** | 合规要求高，AI 代码不可审计 | 设计偏离记录 + 审计日志 + 全链路追溯 |

## 二、教程体系

### 架构：三层漏斗

```
第一层：5分钟体验 ──→ 降低认知门槛，建立第一印象
    │
    ▼
第二层：场景教程 ──→ 解决实际问题，建立信任
    │
    ▼
第三层：深度内容 ──→ 方法论输出，建立权威
```

### 第一层：5 分钟体验

| 内容 | 形式 | 状态 |
|------|------|------|
| **「跟着做」入门指南** | `examples/hello-stdd/` 改造为分步教程，每步有说明 | 📝 待做 |
| **3 分钟演示 GIF** | 录屏：`stdd init` → `/stdd-understand` → `/stdd-continue` → `git tag` 全流程 | 📝 待做 |
| **README 顶部嵌入 GIF** | 用户打开 GitHub 第一眼看到完整演示 | 📝 待做 |

### 第二层：场景教程（3 篇核心）

| # | 教程 | 目标用户 | 核心展示 | 状态 |
|---|------|---------|---------|------|
| 1 | **「用 STDD 写一个 REST API」** | 后端开发者 | Phase 2 spec 编写 → TC-ID 映射 → TDD RED/GREEN/REFACTOR 循环 | 📝 待写 |
| 2 | **「排查一个 AI 跑偏的 Bug」** | AI 编程实践者 | 11 类失败模式实例：展示 `stdd diff` / `stdd trace` 定位问题 | 📝 待写 |
| 3 | **「长程模式：放下电脑去喝咖啡」** | 团队 Leader | Gate 2 后全自动执行全过程，展示 test-report.md 最终产出 | 📝 待写 |

### 第三层：深度内容

| 主题 | 形式 | 目标渠道 |
|------|------|---------|
| 「AI 编程质量不靠运气，靠系统」 | 长文（3000字+） | 掘金 / 知乎 / 公众号 |
| 「11 类失败模式详解」 | 系列文章（3-4篇） | 掘金 / Medium |
| 「AI 编程翻车现场：裸写 vs STDD 对比实验」 | 对比评测文 | V2EX / 掘金 / Reddit |
| 「从 OpenSpec 到 STDD：迁移指南」 | 技术迁移文档 | GitHub Discussions |
| 「量化交易系统如何用 STDD 保证质量」 | 行业案例 | 公众号 / 金融科技社区 |

## 六、关键指标

### 北极星指标

> **周活跃 change 数**：每周有多少用户在 STDD 中启动了新的 change 周期。
> 这个指标代表用户真正在用 STDD 做开发，而非只是浏览。

### 过程指标

| 阶段 | 指标 | 基准（当前） | 1 月目标 | 3 月目标 |
|------|------|-------------|---------|---------|
| 认知 | GitHub Star 数 | 0 | 100 | 500 |
| 认知 | 掘金文章阅读量 | 0 | 5,000 | 30,000 |
| 认知 | 官网 PV（百度统计）| 0 | 2,000/月 | 10,000/月 |
| 兴趣 | README GIF 播放率 | — | 30%+ | 50%+ |
| 兴趣 | 下载按钮点击量 | 0 | 200 | 1,000 |
| 转化 | GitHub Clone 数 | 0 | 80/周 | 300/周 |
| 转化 | `stdd init` 执行量 | 无法追踪 | — | — |
| 留存 | GitHub Issues 提交数 | 0 | 10 | 50 |
| 传播 | 社区贡献者数 | 0 | 3 | 15 |
| 传播 | 自发分享/引用 | 0 | 5 | 30 |
